# Plan 3.5: pulido + repaso básico — diseño

> Origen: feedback del usuario tras validar la app v0.1 (PR #3) en tablet, 2026-07-08.
> Complementa `2026-07-06-dokusho-renshuu-design.md` y el anexo de Plan 3.

## Problema

La v0.1 funciona pero tiene: (1) glosas del diccionario ilegibles (texto concatenado sin separadores), (2) palabras escritas en kana sin definición aunque existen en el db, (3) UI híbrida español/inglés, (4) primera "página" de cada historia es el encabezado `一` de Aozora, (5) cards de biblioteca y lector sin metadata útil, (6) sección "Catálogo" vacía sin propósito visible, (7) pantallas Detalle kanji y Acerca de sin diseño (fondo claro fijo, sin jerarquía), (8) sin ícono de app.

## Decisiones (brainstorming 2026-07-08, aprobadas)

- PR #3 se mergeó como v0.1; todo esto va en branch `feature/plan-3.5-pulido`, PR propio.
- **Idioma: UI toda en inglés.** Las glosas ya son inglesas (Jitendex); coherencia gratis. Código, comentarios y docs del repo siguen en español (convención del monorepo, no visible al usuario).
- **Metadata precalculada en catálogo v2** (no computada en la app).
- **Repaso básico ahora, SRS real en Plan 4**: registrar kanjis consultados + tag manual de dificultad + 3 tarjetas en el home. Sin scheduling.

## Frente A — `diccionario/` (fix de glosas + db-v2)

**Raíz del texto pegoteado**: `jitendex.py` aplana el structured content de Jitendex (tags gramaticales, glosas y oraciones de ejemplo embebidas) en un solo string por glosa.

- Parsear el structured content separando: glosas (lista de strings limpios), descartando tags de PoS embebidos y oraciones de ejemplo inline (los ejemplos ya vienen de Tatoeba por otra vía).
- Tests con fixtures reales del formato Jitendex actual (li anidados, notas, referencias `→`).
- Rebuild → `diccionario-v2.db`, release **db-v2** en GitHub. Tabla `metadata`: `version=2`.
- Actualizar `MAX_ORACIONES_POR_KANJI`/verify_db solo si el tamaño supera 80 MB (palanca ya acordada).
- La app: `VERSION_ESPERADA = 2`, URL del release db-v2, gradle task apunta al asset nuevo.

## Frente B — `historias/` (catálogo v2)

Schema `catalogo.json` v2 — `{"version": 2, "historias": [...]}` con campos nuevos por historia:

```json
{
  "id": "momotaro",
  "titulo": "桃太郎",
  "titulo_lectura": "ももたろう",
  "titulo_en": "Peach Boy",
  "autor": "楠山正雄",
  "dificultad": "facil",
  "tamaño": 67523,
  "version": 2,
  "kanjis_unicos": 124,
  "oraciones": 312
}
```

- `titulo_lectura` y `titulo_en` curados a mano en la config del pipeline (4 historias: Momotarō = "Peach Boy", 浦島太郎 = "Urashima Tarō", 一寸法師 = "The Inch-High Samurai", かちかち山 = "Kachi-Kachi Mountain"). `titulo_en` opcional (null si no hay traducción conocida).
- `kanjis_unicos` y `oraciones` computados por el pipeline al emitir.
- **Filtro de encabezados de sección**: oraciones que son solo numerales japoneses (`一`, `二`, `三`…, opcionalmente con espacios) se eliminan al segmentar (mueren las "primeras páginas ―"). Test con momotaro real.
- Regenerar `catalogo/` completo y commitear. `version` de cada historia sube a 2 (el JSON de historia no cambia de schema; solo re-emisión sin encabezados).
- La app exige `version == 2` en el catálogo (catálogo e historias se regeneran y publican juntos; no hay período de compatibilidad v1). `titulo_en` puede ser null; `titulo_lectura`, `kanjis_unicos` y `oraciones` son obligatorios.

## Frente C — `app/`

### C1. Bug kana sin definición
`BuscadorPalabras.consultar`: orden de lookup pasa a **superficie → formaBase → lecturaHiragana** (del token Kuromoji). El db ya tiene índice `idx_palabras_lectura`. Test: おじいさん (fake con termino=お祖父さん, lectura=おじいさん) resuelve definiciones.

### C2. UI en inglés
Todos los strings de UI a inglés ("Library", "About", "Retry", "No definition", "Previous/Next", "Reading: ", "JLPT (old scale)", "Strokes", "Examples", "Difficulty", etc.). Sin framework i18n (un solo idioma; strings inline como hasta ahora).

### C3. Portada de historia
El lector gana una portada: estado explícito de UI que se renderiza cuando `oracionActual == -1`. Contenido: título grande + `titulo_lectura` como furigana + `titulo_en` + autor + stats ("124 unique kanji · 312 sentences · 37% read"). Botón "Start reading"/"Continue". El progreso guardado sigue apuntando a párrafo/oración reales; la portada se muestra al entrar si no hay progreso, o como página -1 accesible con "Previous" desde la primera oración.

### C4. Cards de biblioteca
Card: título (con furigana chica arriba), `titulo_en` debajo, autor · dificultad, "124 kanji · 312 sentences", barra de progreso si > 0. Grid adaptativo se mantiene.

### C5. Home reestructurado
- Historias remotas no descargadas se integran al mismo grid con botón "Download" en la card (desaparece la sección "Catálogo" separada).
- Error de catálogo remoto: banner chico arriba del grid con "Retry" (no bloquea nada).
- Sección nueva **"Review"** debajo del grid: 3 tarjetas (Easy / Medium / Hard) que muestran cada una un kanji taggeado de esa dificultad (rotando: el consultado más antiguo primero), con kanji grande + lectura principal + primera definición. Tap → Detalle kanji. Si una dificultad no tiene kanjis: tarjeta con hint ("Tag kanji from their detail view"). Si no hay ningún kanji consultado: sección oculta.

### C6. Kanjis consultados + tags
- Tabla Room nueva `kanjis_tocados(kanji TEXT PK, dificultad TEXT NULL, timestamp INTEGER)` — se upsertea en cada apertura de Detalle kanji (registro automático), `dificultad` ∈ {easy, medium, hard} solo si el usuario taggea.
- Migración Room: version 2 con `Migration(1,2)` (CREATE TABLE) — no destruir progreso existente.
- Detalle kanji: fila de 3 FilterChips (Easy/Medium/Hard) — tap taggea/destaggea.
- Insumo directo del generador de mazos de Plan 4 (junto con `palabras_tocadas`).

### C7. Tema + rediseño de pantallas
- Manifest: tema base DayNight (o `android:theme` derivado de M3) para que Detalle kanji y Acerca de respeten dark mode; toda pantalla envuelta en `Surface`/`Scaffold` con colores del tema.
- **Detalle kanji**: layout con jerarquía — kanji gigante centrado, chips de dificultad, secciones con espaciado (Meanings como chips o lista con bullets, On/Kun en filas etiquetadas, JLPT/Strokes como stats compactas, Examples con japonés arriba y traducción gris abajo, separadores entre ejemplos).
- **PalabraSheet**: contenido scrolleable (`verticalScroll`), definiciones como lista numerada (1., 2., …) con espaciado — legibles gracias al Frente A.
- **About**: tipografía con jerarquía (título, secciones con headers, bullets), scrolleable, tema correcto.

### C8. Ícono
`app/logo.jpg` (Saitama) → adaptive icon: generar mipmaps (foreground con el arte + background sólido) en `app/app/src/main/res/mipmap-*`; manifest `android:icon`. El JPG original se mueve a `app/arte/logo.jpg` (fuente). **Solo uso personal** — arte con copyright (ONE/Yusuke Murata), no publicable en stores; nota en README.

## Manejo de errores

- Catálogo v2: parser valida versión == 2 y campos nuevos (tipos); catálogo corrupto → mismo flujo actual (descartar, Retry).
- Migración Room fallida → no hay fallback destructivo (Room lanza; preferible crash visible en debug a perder progreso silenciosamente).
- Tarjetas de Review con kanji que ya no está en el db (db actualizado): tarjeta se salta ese kanji (query defensiva).

## Testing

- JVM/Robolectric como hasta ahora; TDD por task.
- Frente A: tests de jitendex con fixtures del formato real (glosas separadas, sin tags embebidos).
- Frente B: tests de filtro de encabezados + schema v2 (emisor + verify_catalogo actualizado: valida campos nuevos, `version=2`).
- Frente C: tests de lookup por lectura, migración Room 1→2, VM de review cards, parser catálogo v2.
- Validación final en tablet: checklist corto (portada, cards, review section, tags, sheet legible, dark mode en todas las pantallas, ícono).

## Fuera de scope

Scheduling SRS, mazos .apkg, import de texto (Plan 4). Selector de idioma. Búsqueda libre de diccionario.
